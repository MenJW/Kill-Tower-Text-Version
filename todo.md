# Kill Tower Text Version 开发总规划

## 1. 项目目标

目标不是做一个“受杀戮尖塔2启发”的文字卡牌游戏，而是做一个针对指定版本的、纯文字交互版杀戮尖塔2：

- 规则一致：战斗流程、卡牌/遗物/药水效果、敌人行动逻辑、事件结果、地图与奖励公式都要与原版一致。
- 数据一致：角色、卡牌、遗物、怪物、事件、遭遇、机制说明、飞升修正等内容要与目标版本一致。
- 交互不同：不做图形界面，不使用图片、动画和鼠标为核心交互，改为终端文字框 + 键盘操作。
- 架构可追更：因为 STS2 仍在 Early Access，项目必须支持重新导入新版本数据、做差异比对、补全规则改动。

这意味着项目必须先做“版本锁定 + 数据快照 + 可验证规则引擎”，再做界面。先做界面会让项目很快失控。

## 2. 外部资料结论

### 2.1 已确认来源

- 官方 Steam 页面确认了 STS2 的高层特征：5 名角色、单人模式、最多 4 人合作模式、持续更新的高塔内容、故事碎片、Early Access 持续迭代。
- Spire Codex 是一个基于 sts2.dll 反编译与游戏文件提取构建的数据 API/网站，提供了卡牌、角色、遗物、怪物、药水、附魔、遭遇、事件、能力、关键词、意图、球体、减益、修正器、成就、Act、飞升等结构化数据，并支持多语言。
- Kotone Workshop 的 STS2 综合手册也明确说明其数据来自 sts2.dll 反编译与官中本地化，适合作为机制交叉校验来源。

### 2.2 关键发现

- 两个数据源展示出来的实体数量并不完全一致，这很正常，原因通常是版本不同、统计口径不同、某些数据页是否计入测试/隐藏/未完全解析实体不同。
- 因此项目绝对不能把“607 张卡牌”“288 个遗物”之类的数字写死在代码里，所有数量都必须来自快照 manifest。
- Spire Codex 的价值不在网页本身，而在它的结构化数据能力。它公开了成体系的 API 与导出能力，适合作为导入源。
- Kotone Workshop 更适合作为机制核对和中文描述校验来源，而不是运行时依赖。

## 3. 项目总原则

### 3.1 先锁版本，再谈一致性

“与原版完全一致”只能在“与某个明确 build 完全一致”的前提下成立。第一步必须定义：

- 目标游戏版本号
- Steam build id
- 数据抓取日期
- 数据源快照版本
- 该版本下的已知未解析项

建议将第一版目标定义为：

- 单机主流程完全可玩
- 与某个固定 STS2 build 的规则/数据一致
- 可完整打通一局游戏
- 所有可见内容以中文显示，内部逻辑同时保留英文 id

### 3.2 先单机完全一致，再扩展合作

官方页面已经明确 STS2 有最多 4 人合作模式。如果目标是“最终完全一致”，合作模式不能被永久忽略，但绝对不应该进入第一阶段核心开发。

推荐拆分为：

- Phase A：单机完全一致
- Phase B：合作模式一致
- Phase C：版本追更、附加功能、开发工具完善

### 3.3 不直接依赖网页抓取作为运行时逻辑

网页会改版、字段会变、反爬会变。运行时必须只依赖项目内部整理好的标准化数据快照。

正确做法是：

- 导入一次外部数据
- 标准化为本项目 schema
- 固化为本地 snapshot
- 游戏引擎只读取 snapshot

## 4. 推荐技术选型

### 4.1 总体建议

为了尽快做出可玩的版本，建议整个项目统一用 Python。

- 游戏核心：Python 3.12
- 文字界面：Textual
- 终端渲染：Rich
- CLI 管理命令：Typer
- 数据模型：Pydantic
- 高性能 JSON：orjson
- 网络请求：httpx
- 存档/索引：SQLite
- 测试：pytest

### 4.2 为什么选 Textual

你的目标是“只有文字框组成的游戏界面”，Textual 比 curses 更适合快速搭建一个结构化 TUI：

- 更容易做多面板布局
- 更容易处理键盘事件
- 更容易做弹窗、日志区、状态栏、列表、表格
- Python 全栈一致，减少跨语言通信和状态同步成本

### 4.3 为什么不建议首发使用 Python + Web 前端分离

如果首发就拆成 Python 核心 + TypeScript 前端，会增加：

- 前后端状态同步成本
- 调试复杂度
- 命令协议设计负担
- 文本 UI 与规则引擎分层工作量

除非你确定后续一定要发布浏览器版，否则第一阶段没有必要。

## 6. 核心架构设计

### 6.1 分层原则

必须明确区分以下 5 层：

1. 原始数据层：外部来源的 JSON/页面/导出文件。
2. 标准化内容层：本项目统一 schema 下的卡牌、遗物、怪物、事件等。
3. 规则引擎层：战斗、地图、事件、商店、奖励等纯逻辑。
4. 表现层：Textual 终端界面。
5. 调试与验证层：回放、日志、差异测试、回归测试。

如果把内容数据、规则逻辑、UI 文本混在一起，后面会很难追版本。

### 6.2 数据层

数据层负责把 Spire Codex / Kotone 资料整理为统一格式，建议实体至少覆盖：

- characters
- cards
- relics
- potions
- monsters
- encounters
- events
- powers
- keywords
- intents
- orbs
- afflictions
- modifiers
- acts
- ascensions
- achievements

每个实体都应包含：

- 稳定内部 id
- 英文原始 id
- 中文显示名
- 所属版本信息
- 原始来源元数据
- 渲染所需字段
- 逻辑所需字段
- 是否需要脚本层实现的标记

### 6.3 规则引擎层

规则引擎不能只靠“配置表”解决，因为 STS2 中一定存在大量需要脚本表达的动态效果。

推荐结构：

- ActionQueue：所有效果都排队执行，确保结算顺序稳定。
- EffectContext：提供当前战斗状态、操作者、目标、随机数、来源对象。
- TriggerBus：统一处理回合开始/结束、出牌、弃牌、抽牌、受伤、死亡等触发点。
- RuleResolver：负责判定效果合法性、目标选择、动态数值结算。
- Registry：按 card_id / relic_id / potion_id / monster_id / event_id 绑定具体实现。

### 6.4 为什么必须有脚本注册表

外部数据通常能给出：

- 名称
- 描述
- 数值变量
- 稀有度
- 类型
- 目标类型

但不能天然解决所有复杂行为，例如：

- 依赖上下文的随机生成卡牌
- 连锁触发
- 特殊事件分支
- 多阶段敌人行为
- 临时能力与永久能力交互
- “本回合第一次”“每场战斗一次”“如果上回合发生 X”这类条件效果

因此必须建立脚本注册表，把静态数据和动态行为分开。

### 6.5 文本界面层

UI 只能做两件事：

- 展示状态
- 把用户选择翻译成引擎命令

UI 不应该：

- 自己计算规则
- 自己持有一份独立游戏状态
- 写任何数值平衡逻辑

## 7. 数据导入与标准化流程

### 7.1 推荐导入策略

优先使用 Spire Codex 的结构化导出能力作为主来源，Kotone 资料作为校验来源。

标准流程：

1. 选择目标 build。
2. 拉取 Spire Codex 导出数据，至少获取 eng 与 zhs。
3. 保存原始快照到 data/raw 与 data/snapshots。
4. 生成 manifest.json，记录版本号、build id、抓取时间、实体数量、来源 URL。
5. 执行 normalize，把不同来源字段统一成项目内部 schema。
6. 生成 gap report，列出哪些实体已完整支持、哪些只有数据未实现逻辑。
7. 用 Kotone 的机制页做二次人工校验，尤其是事件数值、商店价格、奖励公式、飞升修正。

### 7.2 manifest 建议字段

```json
{
	"game": "Slay the Spire 2",
	"app_id": 2868840,
	"game_version": "x.y.z",
	"build_id": "xxxxx",
	"snapshot_tag": "2026-03-xx_build_xxxxx",
	"fetched_at": "2026-03-xxT00:00:00Z",
	"sources": [
		"https://github.com/ptrlrd/spire-codex",
		"https://kotoneworkshop.com/sts2-builds"
	],
	"counts": {
		"characters": 0,
		"cards": 0,
		"relics": 0,
		"monsters": 0,
		"encounters": 0,
		"events": 0
	}
}
```

### 7.3 标准化时必须解决的问题

- 同一实体在不同来源的命名差异
- 中文名与英文 id 的映射
- 动态数值字段的统一表示
- 事件分支的结构化表示
- 怪物行动模式的可执行表示
- 升级卡与原始卡的关联
- 起始遗物升级链或替换链
- 关键词、能力、状态效果的唯一 id 约束

## 8. 游戏系统拆解

### 8.1 跑局主循环

必须实现的最小闭环：

- 主菜单
- 开始新游戏
- 选择角色
- 设定种子或随机种子
- 生成地图
- 进入房间
- 战斗 / 事件 / 商店 / 篝火 / 宝箱 / 精英 / Boss
- 领取奖励
- 进入下一层
- Act 切换
- 胜利 / 死亡结算
- 存档与继续

### 8.2 战斗系统

战斗系统是整个项目最重要的部分，建议按以下子系统实现：

- 回合开始、玩家阶段、敌人阶段、回合结束
- 能量、抽牌、弃牌、洗牌、耗尽、放逐等区域管理
- 攻击、格挡、易伤、虚弱、脆弱等核心状态
- 目标选择、随机目标选择、多段攻击
- 临时属性、永久属性、层数型能力、计数型能力
- 召唤、死亡、复活、转阶段
- 出牌限制、费用变化、X 费牌、免费牌
- 生成卡牌、复制卡牌、临时卡牌、升级卡牌
- 战斗奖励结算

### 8.3 卡牌系统

卡牌需要区分三层：

- CardDefinition：静态内容定义
- CardInstance：实际战斗中的实例，含升级、临时 cost、附加状态
- CardEffectScript：具体效果实现

关键难点：

- 动态描述数值与实际结算保持一致
- 升级前后字段差异
- 牌堆中实例状态持久化
- 生成牌是否继承升级与临时修正

### 8.4 遗物系统

遗物需要完整覆盖触发时机：

- 战斗开始
- 回合开始/结束
- 使用卡牌时
- 造成伤害/受到伤害时
- 击杀敌人时
- 进入房间时
- 获得奖励时
- 商店/事件/篝火交互时

建议为遗物设计统一 hook 接口，而不是在战斗代码里写大量 if。

### 8.5 药水系统

药水要支持：

- 指定目标 / 无目标 / 全体目标
- 战斗中使用 / 战斗外使用的区别
- 与遗物、能力、飞升修正的联动
- 获得、替换、丢弃、上限管理

### 8.6 怪物与遭遇系统

怪物系统至少要分为：

- MonsterDefinition：基础属性、行动列表、数值区间、标签
- MonsterInstance：当前战斗中的生命、状态、阶段、记忆状态
- MonsterBehaviorScript：下一步意图生成逻辑
- EncounterDefinition：房间类型、Act、怪物组合、特殊规则

难点在于：

- 多怪协同
- 首回合特殊行动
- 周期性行动表
- 条件分支行动
- 阶段切换
- 飞升数值差异

### 8.7 地图与房间系统

地图系统需要支持：

- Act 地图生成
- 节点连接规则
- 房间类型分布
- 精英/Boss/宝箱/商店/休息点约束
- 路线选择
- Act Boss 池选择逻辑

### 8.8 事件系统

事件是文字版项目里最容易“看起来简单，实际很难”的部分，因为它们大量依赖：

- 页面切换
- 选项条件
- 动态数值
- 角色限制
- 奖励/惩罚复合效果
- 持续性后果

建议把事件系统建模为：

- EventDefinition
- EventPage
- EventChoice
- EventOutcome
- EventScript

页面层负责显示，Outcome 只描述结果类型，真正执行由脚本层完成。

### 8.9 商店系统

商店不能只做一个“买东西菜单”，必须对齐机制：

- 卡牌价格公式
- 遗物价格区间
- 药水价格区间
- 移除卡牌的价格递增
- 假商人或特殊商店事件
- 刷新逻辑与库存构成
- 折扣、加价、遗物影响

### 8.10 篝火与奖励系统

要覆盖：

- 休息
- 升级
- 特殊角色或遗物解锁的额外选项
- 宝箱开箱
- 战斗后卡牌奖励
- 金币、药水、遗物掉落公式
- Boss 奖励

### 8.11 飞升与修正系统

如果要做到“原版一致”，飞升不是可选项，而是核心系统之一。

必须支持：

- 飞升等级配置
- 各等级修正内容
- 敌人属性变化
- 奖励变化
- 特定规则变化

### 8.12 多人合作系统

合作模式不建议进入首发，但要提前为它预留架构：

- 模拟层必须支持多个玩家实体
- 指令层必须支持多输入源
- 回合同步必须是权威状态机驱动
- 所有随机数必须由 host 或权威模拟器统一生成

如果未来做合作，建议采用：

- server-authoritative 模式
- 文本命令同步
- 全局种子 + 操作日志回放

## 10. 详细开发阶段

### Phase 0：需求冻结与版本基线

目标：定义“完全一致”到底对齐哪个版本。

任务：

## 11. 测试与验收策略

### 11.1 必须有的测试类型

- 单元测试：基础公式、状态变化、目标选择、队列顺序。
- 集成测试：完整战斗、事件链、商店购买、奖励发放。
- 回归测试：固定 seed 的标准跑局片段。
- Golden Transcript：把战斗日志作为金标，确保重构后文字结果与逻辑结果一致。
- 数据校验测试：缺字段、断引用、重复 id、无效本地化。
- 模糊测试：随机生成一批对战或跑局，找崩溃与死循环。

### 11.2 强烈建议增加的验证工具

- replay 文件：记录每一步操作与随机数。
- debug 面板：显示当前 hook、队列、状态层数。
- parity runner：跑一组标准场景并输出差异。
- entity gap report：按卡牌/遗物/事件统计是否已实现。

### 11.3 验收标准

首发单机版至少应达到：

- 目标 build 下，5 名角色可创建新局。
- 战斗、地图、事件、商店、篝火、宝箱、Boss、奖励可完整运行。
- 中文文本可读，英文 id 可用于 debug。
- 同一 seed 可稳定重现。
- 存档、读档、回放可用。
- 已知未实现项被明确列出，不存在“静默错误”。

## 12. 风险清单与应对

### 风险 1：Early Access 版本变化过快

应对：

- 每个版本都做 snapshot。
- 所有逻辑以 snapshot tag 为边界。
- 提供 diff 工具，不直接覆盖旧数据。

### 风险 2：部分复杂效果无法纯数据驱动

应对：

- 采用脚本注册表。
- 先做高频公共效果，再补个体特殊逻辑。
- 用 gap report 防止遗漏。

### 风险 3：中文描述正确但实际数值不一致

应对：

- 文本描述永远从实际 effect 参数生成或校验。
- 为关键卡牌和事件建立数值回归测试。

### 风险 4：项目在 UI 上过度消耗时间

应对：

- UI 只做显示与输入映射。
- 先做 playable vertical slice，再做视觉优化。

### 风险 5：合作模式把单机版本拖垮

应对：

- 单机先发布。
- 一开始就做可扩展状态模型，但不提前做网络层。

### 风险 6：版权与分发边界不清

应对：

- 不把官方图像、音频、动画当成必需资产。
- 优先分发脚本、逻辑、标准化数据工具，必要时要求用户自行导入合法获得的数据快照。

## 13. 推荐开发顺序

如果是单人开发，最稳的顺序是：

1. 版本冻结。
2. 数据导入与 schema。
3. 战斗内核。
4. 一个角色 + 一批常见敌人的可玩垂直切片。
5. 文本战斗 UI。
6. 地图/事件/商店/篝火。
7. 完整单机跑局。
8. 其余角色与内容补完。
9. 飞升与高级系统。
10. 合作模式。

不要反过来做。先做“完整 UI”或“全量内容抓取”都会把项目节奏拖垮。

## 15. 最终交付标准

当以下条件都成立时，可以认为项目达到了“纯文字版 STS2”的核心目标：

- 使用固定 snapshot 能稳定运行完整单机流程。
- 文本界面可独立完成整局游戏。
- 规则与数值与目标版本一致，且有测试可证明。
- 数据更新有自动化流程，不依赖手工改表。
- 所有未实现项都被显式报告。
- 合作模式被列为后续扩展而不是阻塞单机版发布。

## 16. 结论

这个项目真正的难点不是“把界面做成文字”，而是“把 STS2 的内容与规则拆成可追更、可验证、可维护的模拟器”。

因此正确路线是：

- 先做版本快照
- 再做规则引擎
- 再做文字界面
- 再做全量内容补完
- 最后做合作模式

如果严格按照这个框架推进，这个项目是可以落地的；如果一开始就追求“全量内容 + 全功能 + 全界面”，大概率会在中途失控。
